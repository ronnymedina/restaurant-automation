Capas de Protección contra Ingeniería Inversa

El problema real de Electron: app.asar es trivialmente extraíble. Estas son las defensas.

⚠️ El problema: app.asar es público por defecto
asar extract app.asar ./output → código fuente legible al instante
Capa 1 — crítica
🔴 V8 Bytecode con bytenode
Compila los archivos .js de NestJS a bytecode de V8 (.jsc). El resultado es binario ilegible — no hay decompilador público que lo revierta. Es la protección más fuerte disponible sin herramientas comerciales.
bytenode --compile dist/main.js → dist/main.jsc (binario V8)
Efectividad
Capa 2 — alta
🟡 Ofuscación con javascript-obfuscator
Para los archivos de entrada (main.js, preload, partes que bytenode no puede compilar). Activa: control flow flattening, string encryption, dead code injection, renombrado agresivo de variables.
javascript-obfuscator main.js --compact true --control-flow-flattening true --string-array-encoding base64
Efectividad
Capa 3 — media
🟢 Electron Fuses — deshabilitar herramientas de debug
Electron tiene "fuses" (switches de compilación) que se aplican al binario antes de distribuir. Deshabilitan DevTools, el protocolo de debugging remoto y las flags de Node.js en CLI — impide que alguien conecte un debugger al proceso.
@electron/fuses: DevTools=off, NodeCliInspect=off, RunAsNode=off
Efectividad
Capa 4 — integridad
🔵 ASAR integrity + Code Signing
Electron 30+ tiene verificación de integridad del ASAR integrada. El binario firmado (Authenticode en Win, Apple Developer en Mac) verifica que el app.asar no fue modificado al arrancar. Si alguien parchea el código, la firma falla.
electron-builder: asarUnpack mínimo, enableASARIntegrity: true, firmado con certificado
Efectividad
Capa 5 — la protección real
🟣 Validación criptográfica del servidor (RSA)
Aunque alguien rompa todas las capas anteriores y parchee la validación local — no puede generar tokens RSA válidos sin tu clave privada. El servidor de licencias es la única barrera que no se puede evadir localmente. Esto es lo que protege tu negocio.
Token firmado con RSA-256 private key (solo en tu servidor) → verificado con public key embebida
Efectividad
⚡ Realidad del mercado de restaurantes
Con las capas 1 + 3 + 5, el esfuerzo para crackear tu software supera ampliamente el costo de comprar la licencia. Tu cliente objetivo (dueño de restaurante) no tiene ni el conocimiento ni el incentivo para hacer ingeniería inversa a un software de US$200–500. Las protecciones son suficientes para ese mercado — no necesitas el nivel de protección de Adobe o Microsoft.
