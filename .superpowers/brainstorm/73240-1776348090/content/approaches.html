3 enfoques para unificar el frontend

Selecciona el que más se alinee con tus prioridades

A

Una sola app Astro — static + React islands ★ Recomendado

output: 'static' + @astrojs/react. NestJS sirve los archivos estáticos y maneja un wildcard fallback para rutas dinámicas como /kiosk/[slug]. SSE reemplaza WebSockets.

A favor

  • Un solo proceso en Electron (NestJS + static files)
  • Build pipeline ya existe (copy-static.mjs)
  • Componentes React reutilizables entre kiosk y dash
  • URLs limpias (/kiosk/mi-restaurante)
  • Menos dependencias — sin adaptador Node en Astro

En contra

  • NestJS necesita configurar ServeStaticModule + fallback SPA
  • Sin SSR — SEO de homepage depende de prerender estático
Flujo: astro build → copy-static.mjs → api-core/public/ → NestJS sirve todo
B

Astro hybrid — páginas estáticas + kiosk SSR

output: 'hybrid': dashboard y homepage se prerenderizan, kiosk es server-rendered. Para cloud funciona perfecto. Para Electron necesita dos procesos (Astro server + NestJS binary).

A favor

  • Kiosk puede tener lógica server-side si se necesita en el futuro
  • No requiere cambios en NestJS para routing

En contra

  • Electron necesita spawnear 2 procesos (complejo)
  • Contradice el objetivo de simplicidad desktop
  • Más configuración para coordinar puertos entre servicios
Flujo Electron: spawn NestJS binary + spawn Astro server → abrir browser
C

Dos apps — marketing separado del app

apps/ui-marketing (landing, ultra-liviano, CDN) + apps/ui-app (dashboard + kiosk, React). Build guide ya mencionaba esta estructura.

A favor

  • Landing puede deployarse en CDN independiente (Vercel free tier)
  • Separación de concerns — marketing no afecta el app

En contra

  • Dos apps para mantener (duplicación de config, deps)
  • No hay componentes compartidos fácilmente
  • Más mantenimiento para una sola persona
Complejidad: más alto vs beneficio marginal para este stage del proyecto
